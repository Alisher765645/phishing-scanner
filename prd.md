# PRD: Phishing Scanner

**Версия:** 1.0
**Дата:** 01.07.2026
**Статус:** Draft
**Автор:** —

---

## 1. Обзор продукта

Phishing Scanner — инструмент для быстрой эвристической проверки email-писем
на признаки фишинга. Пользователь вставляет текст письма (в идеале — с
заголовками) или загружает `.eml`-файл, инструмент анализирует заголовки,
ссылки и текст письма и возвращает риск-скор (0–100) с расшифровкой
конкретных найденных признаков.

Продукт состоит из двух интерфейсов поверх общего движка анализа:

| Компонент | Назначение |
|---|---|
| `phishing_analyzer.py` | Общий движок детекции — единственный источник логики |
| Telegram-бот | Быстрая проверка прямо в мессенджере |
| Web-приложение | Проверка в браузере с визуализацией результата |

---

## 2. Проблема

Обычный пользователь получает подозрительное письмо и не имеет быстрого
способа оценить, фишинг это или нет, кроме интуиции. Специализированные
корпоративные средства защиты почты (SEG, sandboxing) недоступны частным
пользователям и малым командам. Нужен лёгкий инструмент «вставил письмо —
получил понятный вердикт», не требующий регистрации и внешних API.

---

## 3. Цели

- Дать пользователю понятный риск-скор и объяснение находок за секунды.
- Работать без сторонних сервисов и хранения данных — вся логика локальна.
- Быть достаточно простым, чтобы им могли пользоваться нетехнические люди.

**Не цели (out of scope для v1):**
- Замена корпоративных Secure Email Gateway.
- 100%-ное обнаружение (это эвристика, не ML-классификатор и не live
  threat-intel).

### Метрики успеха
- Корректная классификация на наборе тестовых писем (ручных примеров):
  0 ложных срабатываний на «чистых» письмах, риск-скор ≥ 60 на явных
  фишинговых образцах.
- Время ответа < 1 сек на письмо (без внешних API-вызовов).

---

## 4. Целевая аудитория

- Частные пользователи, получающие подозрительные письма.
- Небольшие команды/малый бизнес без выделенного security-отдела.
- Технически любопытные пользователи, которые хотят держать бота под рукой
  в Telegram.

---

## 5. Область действия (Scope)

### В скоупе (v1)
- Анализ заголовков: From / Reply-To / Return-Path, SPF/DKIM/DMARC (если
  присутствуют в `Authentication-Results`), display-name spoofing.
- Анализ ссылок: shortener'ы, IP-адреса вместо домена, punycode,
  несоответствие anchor text и href, подозрительные TLD, отсутствие HTTPS,
  похожие на бренды домены.
- Анализ текста: формулировки срочности/давления, запросы конфиденциальных
  данных, обезличенные приветствия.
- Скоринг 0–100 и вердикт (🟢/🟡/🟠/🔴).
- Telegram-бот: команды `/start`, `/help`, приём текста и `.eml`/`.txt`.
- Web-приложение: форма ввода + визуализация (радиальный индикатор риска,
  разбивка находок по severity, список ссылок).

### Вне скоупа (v1, кандидаты для v2)
- Проверка репутации домена через внешние API (Google Safe Browsing,
  VirusTotal).
- Хранение истории проверок / база данных.
- Аутентификация и мультипользовательский режим.
- ML-модель классификации.
- Экспорт отчёта в PDF.
- Браузерное расширение.

---

## 6. User Stories

1. **Как пользователь**, я хочу вставить текст письма и получить риск-скор,
   чтобы быстро понять, доверять ли письму.
2. **Как пользователь**, я хочу загрузить `.eml`-файл, чтобы не копировать
   заголовки вручную.
3. **Как пользователь**, я хочу видеть *почему* письмо помечено как
   подозрительное (конкретные находки), а не только цифру.
4. **Как пользователь Telegram**, я хочу получить отчёт прямо в чате, не
   открывая браузер.
5. **Как пользователь веб-версии**, я хочу визуально считывать уровень риска
   (цвет/индикатор), не вчитываясь в текст, если я тороплюсь.

## 7. Функциональные требования

### 7.1 Движок анализа (`phishing_analyzer.py`)

| ID | Требование |
|---|---|
| FR-1 | Принимает сырой текст письма (с заголовками) или произвольный текст |
| FR-2 | Определяет, является ли вход «сырым письмом» (по наличию `From:`/`Subject:`/`To:`) |
| FR-3 | Извлекает и сравнивает домены From / Reply-To / Return-Path |
| FR-4 | Проверяет display-name на совпадение с известными брендами при несовпадающем домене |
| FR-5 | Парсит `Authentication-Results` на предмет fail по SPF/DKIM/DMARC |
| FR-6 | Извлекает все ссылки (plain text + HTML `<a href>`) |
| FR-7 | Флагирует IP-ссылки, punycode, shortener'ы, подозрительные TLD, не-HTTPS, brand-lookalike домены, mismatch anchor/href |
| FR-8 | Флагирует формулировки срочности, запросы credential'ов, обезличенные приветствия |
| FR-9 | Считает взвешенный риск-скор (high=25/medium=12/low=5, cap 100) и вердикт по порогам (0 / 1–29 / 30–59 / 60–100) |

### 7.2 Telegram-бот

| ID | Требование |
|---|---|
| FR-10 | `/start` и `/help` выводят инструкцию по использованию |
| FR-11 | Принимает произвольный текст сообщением — анализирует |
| FR-12 | Принимает `.eml`/`.txt` документом — скачивает, декодирует, анализирует |
| FR-13 | Отвечает форматированным сообщением: вердикт, скор, список находок с иконками по severity, до 10 найденных ссылок |
| FR-14 | Токен бота берётся из переменной окружения `TELEGRAM_BOT_TOKEN`, никогда не хардкодится |

### 7.3 Web-приложение

| ID | Требование |
|---|---|
| FR-15 | `GET /` отдаёт страницу с формой (textarea + file input) |
| FR-16 | `POST /api/analyze` принимает текст (JSON) либо файл (multipart), возвращает JSON-результат |
| FR-17 | Frontend отрисовывает радиальный SVG-индикатор риска, окрашенный по уровню (зелёный→жёлтый→оранжевый→красный) |
| FR-18 | Frontend отрисовывает разбивку находок по severity (high/medium/low) в виде баров |
| FR-19 | Frontend показывает список находок с точками, окрашенными по severity |
| FR-20 | Frontend показывает список извлечённых ссылок (до 10, с указанием общего числа) |

---

## 8. Нефункциональные требования

| Категория | Требование |
|---|---|
| Приватность | Никаких внешних сетевых вызовов при анализе; текст письма не сохраняется на сервере |
| Безопасность | Инструмент не переходит по ссылкам и не исполняет содержимое письма — анализируется только текст |
| Производительность | Ответ `/api/analyze` — до 1 секунды для писем ≤ 20 000 символов |
| Совместимость | Web-версия — актуальные Chrome/Firefox/Safari, адаптивна до мобильных экранов (breakpoint 860px) |
| Надёжность | При отсутствии заголовков в письме — деградация до анализа текста/ссылок без ошибки |
| Конфигурация | Секреты (токен бота) — только через переменные окружения |

## 9. Архитектура

```
Пользователь
   │
   ├─ Telegram ──▶ bot.py ──┐
   │                        │
   └─ Браузер ──▶ app.py ───┼──▶ phishing_analyzer.analyze_email()
        (Flask,               │        │
         /api/analyze)        │        ▼
                               │   {score, verdict, flags[], links[]}
                               │        │
                               ▼        ▼
                        Telegram-сообщение   JSON → SVG-гейдж, бары, списки
```

Общий модуль `phishing_analyzer.py` — единственная точка правды по логике
детекции; оба интерфейса — тонкие обёртки над ним.

## 10. UX / визуальные требования (web)

- Тёмная тема: фон `#0b0f14`, панели `#131a22`, разделители `#232d38`.
- Цвета риска: high `#f0555a`, medium `#f5a623`, low `#e8d44d`, safe `#34c97a`,
  акцент `#4fa3f7`.
- Моноширинный шрифт для цифр/скора (ощущение «терминала безопасности»),
  системный sans-serif — для текста находок.
- Единственный акцентный элемент — радиальный индикатор риска (signature
  element), остальной интерфейс — сдержанный.
- Пустое состояние до первой проверки: подсказка «Результат появится здесь
  после проверки».

---

## 11. Риски и ограничения

- **Ложные срабатывания**: маркетинговые письма с формулировками срочности
  («ограниченное предложение») могут получить завышенный скор.
- **Ложноотрицательные**: продуманный фишинг без типичных паттернов не будет
  пойман — инструмент не заменяет репутационные проверки и ML-детекцию.
- **Нет live-проверки репутации домена** — намеренное решение v1 (приватность
  + отсутствие внешних зависимостей), компенсируется в roadmap.
- **Открытый эндпоинт**: `/api/analyze` без аутентификации — приемлемо для
  локального/личного использования, требует добавления auth при публичном
  деплое.

---

## 12. Roadmap (v2+)

1. Интеграция репутации домена (Google Safe Browsing / VirusTotal API) как
   дополнительный сигнал к эвристикам.
2. История проверок (SQLite) для веб-версии.
3. Экспорт отчёта в PDF.
4. Аутентификация для многопользовательского/командного деплоя.
5. Браузерное расширение (анализ письма прямо в Gmail/Outlook Web).
